    xInjection 


    Introduction

    Powerful IoC library built on-top of InversifyJS inspired by NestJS's DI.

    The xInjection library has been heavily inspired by how NestJS handles its Dependency Injection.

    It is mainly a wrapper of the (battle-tested Inversion of Control library) InversifyJS, allowing to encapsulate with even more granularity the dependencies by using modules derived from the ProviderModule class.

    A ProviderModule allows to import and export providers exactly like how it is done within a NestJS application.

    Each ProviderModule manages its own Container allowing you to easily decouple an implementation by specifying which providers should be exported when another ProviderModule imports a module.

    When using the xInjection library, by default your application will already have a ProviderModule named AppModule, as its name implies, it is a global module which can be used to register dependencies which should be easily injected from anywhere in your application.

    Its default InjectionScope is set to register all the providers as singletons.

    All modules automatically inherit the dependencies registered into the global AppModule!

    Quick Start

    Library Dependencies

    Depending on your environment you may have to first install the reflect-metadata lib.

    npm i reflect-metadata

    TypeScript Configuration

    In your tsconfig.json file add the emitDecoratorMetadata and preserveConstEnums options.

    {
      "compilerOptions": {
        "experimentalDecorators": true,
        "emitDecoratorMetadata": true
      }
    }

    Bootstrapping

    In your application entry-point, you must import the global AppModule and register it:

    import { AppModule, Injectable } from '@adimm/x-injection';

    AppModule.register({});

    Even if you don't plan to have any global provider, you still must invoke the AppModule.register method!

    Providing an empty object {} as the options to the method is accepted and it'll work.

    If you need to provide some global providers which must be resolved from anywhere within your app, now is the perfect place and time to register them:

    @Injectable()
    class ConfigService {
      constructor(private readonly logger: LoggerService) {}
    }

    AppModule.register({
      providers: [LoggerService, ConfigService],
    });

    From now on, you can freely inject the LoggerService and ConfigService anywhere in your app, they'll also be available inside all the ProviderModules floating around your app.

    Custom ProviderModules

    Now that we've seen how we can register the global AppModule, we can start to create a couple of custom modules which we can import around when needed.

    import { Injectable, InjectionScope, ProviderModule } from '@adimm/x-injection';

    @Injectable()
    export class DatabaseService {
      // Big implementation here
    }

    @Injectable()
    export class SessionService {
      constructor(public readonly userService: UserService) {}

      // Big implementation here
    }

    export const DatabaseModule = new ProviderModule({
      // It is optional to provide a name, but it may come in handy when debugging or during unit-tests.
      name: 'DatabaseModule',
      providers: [DatabaseService],
      // e.g: ModuleA imports the DatabaseModule, now the internal container of ModuleA will also hold a reference to the resolved databse service
      // meaning that you can inject the `DatabaseService` from the ModuleA as well.
      exports: [DatabaseService],
      // The optional `onReady` method will be invoked once when the module container has been initialize and the providers resolved.
      // Useful when you have to do additional checks.
      onReady: async (module) => {
        // The `module` variable is a reference to the `DatabaseModule` itself, meaning that you can inject
        // into this scope any available provider, eg:

        const databaseService = module.get(DatabaseService);
      },
      // The optional `onDispose` method will be invoked whenever the internal module dispose method is invoked.
      // Useful when you have to perform a clean-up.
      onDispose: async (module) => {
        // As the `onDispose` method is invoked _before_ the actual dispose process, you
        // can still successfully inject into this scope any available provider, eg:

        const databaseService = module.get(DatabaseService);

        databaseService.closeConnection();
      },
    });

    export const SessionModule = new ProviderModule({
      name: 'SessionModule',
      // We can also change the default scope for the entire module,
      // by default falls back to `Singleton` if not provided.
      defaultScope: InjectionScope.Request,
      providers: [SessionService],
      // Because the scope is set to `Request`, and the `SessionService` provider does not
      // specify a scope itself, it means that whenever a module imports this module
      // it'll receive a new instance of the SessionService.
      exports: [SessionService],
    });

    // Now, if we go back at our bootstrapping step, we could provide those 2 modules into the `AppModule`
    // so they become globally available:

    AppModule.register({
      imports: [DatabaseModule, SessionModule],
    });

    From now on, the AppModule container has the references of the DatabaseService and the SessionService. But it is important to keep in mind of some aspects:

    • We know that the DatabaseService scope is set to Singleton.
    • We know that the SessionService scope is set to Request.

    This means that if we do:

    const databaseService_a = AppModule.get(DatabaseService);
    const databaseService_b = AppModule.get(DatabaseService);

    expect(databaseService_a).toBe(databaseService_b);

    Will produce true, but, doing:

    const sessionService_a = AppModule.get(SessionService);
    const sessionService_b = AppModule.get(SessionService);

    expect(sessionService_a).toBe(sessionService_b);

    Will produce false because Inversify's Request scope acts as a Singleton scope within the same module context and as a Transient scope outside the module context.

    You can also inject multiple dependencies at once by using the getMany method of the module.

    const [serviceA, serviceB] = BigModule.getMany(ServiceA, ServiceB);
    // or
    const [serviceC, serviceD] = BigModule.getMany<[ServiceC, ServiceD]>(SERVICE_TOKEN, 'SERVICE_ID');

    With the above simple examples you should have the basics of how x-injection works 😎.

    Feel free to open an issue if you find a bug or have a question 😄.

    Good to Know

    A ProviderModule instance uses the IProviderModule interface which exposes only the most important methods/properties in order to simplify the usage of a module.

    However, a ProviderModule instance can be casted to the IProviderModuleNaked interface either by using the as keyword, or via the .toNaked() method. As you can see, the IProviderModuleNaked exposes a lot more advanced methods and properties which can be used to enhance the development process in the case when you need a more advanced solution.

    All the IProvderModuleNaked methods which start with __ (double underscore) are wrappers methods of their InversifyJS counterpart, also offering support for the native xInjection ProviderToken type and so on.

    For even more advanced cases, you can access the underlying Inversify Container via the IProviderModuleNaked.container property!

    You can also access the InversifyJS container of the AppModule via the IProviderModuleNaked.container property, or by importing the GlobalContainer

    import { AppModule, GlobalContainer } from '@adimm/x-injection';

    // Or via the `AppModule`.
    const globalContainer = AppModule.toNaked().container;
